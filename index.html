請描述一件產生明顯負面情緒的經歷，如何處理該情緒？

在森子工作室就職時，我在正式拍攝受訪者訪談前，會先研究事前訪談的內容與專案的需求，並以期待得到的答案設計訪問題目與安排題目順序，但是，有時因為客戶的信任感是建立在老闆（也是導演）身上，所以他偶爾也會待在現場參與訪談；當兩個導演同時在現場，有時候因為兩人預設的答案方向不完全相同，就會產生都想引導受訪者的職務重疊狀況，我也會因此有種計畫被打亂的感覺。

我曾經為了職務分配的問題感到苦惱，思考是否有必要與老闆溝通此情況，但仔細回想，其實受訪者的時間大多有彈性能夠接受更多的提問，且經過剪輯確認受訪內容後，我認為以結果來說，兩方的做法儘管方向不同，但都能產生好的結果，既然對於專案的成果都是好的，那我應該做到的是將重心放在現場的變化，並且把專案的成果確實完成客戶的需求放在最重要的順位，而不是我原有的計畫；當確認最重要的事情是什麼後，便會發現先前的苦惱不再出現。

關於這份申請網頁，分享一個開發時的技術心得。

在開發這份申請網頁時，已透過網路資源對 CSS、HTML、JavaScript 有粗淺的了解，但直到自己實作申請網頁時，才注意到有相似功能的 id 跟 class 屬性，有重疊的功能但刻意區分成兩種屬性應該會有不同的使用情境，便更深入地去理解兩個的差異，如 id 是唯一的，在 JavaScript 中使用 getElementById() 中引用；而 class 能夠分組（不是唯一），在 JavaScript 中用 getElementsByClassName() 引用，還有一種方法 querySelectorAll() 兩者都能夠引用；另外，在 CSS 中會以 id 設定的樣式為優先。

其他想要對我們說的事情？

在決定轉職為軟體工程師前，我一直以成為劇情片編導為目標，但在準備電影研究所時，透過書寫自傳、創作劇本等過程，我仔細疏理自己是否要繼續投入這份產業，最後歸納出兩點原因讓我決定轉職：


  • 自我迭代慢

一部劇情短片從寫劇本、籌措資金（輔導金等）、拍攝、後製、參加影展播映，整個過程大約兩年起跳，同時，長期一直關注同一個作品難以客觀地反省自己並作出調整，但我更希望能從事不斷學習並不斷自我迭代的職業。


  • 興趣不一定適合當作職業

當興趣作為職業時，就必須思考如何從中找到商業價值，用興趣謀生，也需要面對社會的期許、他人的肯定，比如報考電影所，其實是源於謀生的恐懼（希望未來從事教職），而寫劇本時需要考慮輔導金的評審喜好，反而遠離原先興趣的純粹。


這兩點促使我開始摸索轉職的方向，在摸索的過程中，我閱讀了《越工作越自由》、《別做熱愛的事，要做真實的自己》等轉職書籍，從中統整出職涯轉換最重要的能力 ── 找出自己的核心能力，並擅用自己的核心能力探索符合自己價值觀的未知領域 ── 我運用此方法摸索出軟體工程師會是一個適合我的選擇，以下幾點分析為何我適合轉職為軟體工程師：


  • 樂於自學也擅長自學

雖然是電影本科系，但實際上大部分的技能還是必須自學，因此我已習慣處於自己學習、分析、歸納、輸出的狀態，比如說過去發現不夠理解編劇技巧時，我便會選擇幾部電影一個場次一個場次研究，並在筆記本上寫下每一場的大綱、人物關係圖等；這段經驗幫助我打開對於編劇的理解，也獲得了學習的成就感。

現在回看便會發現「自學」與「學習的成就感」所達到的正向循環，其實是讓我透過四年學習就入圍金鐘獎的關鍵，而我相信這份自學能力與樂於自學的個性非常適合擔任軟體工程師。


  • 喜歡對事情追根究柢，擅長分析、釐清事情的底層邏輯

在從事影像工作的期間，我就發現自己比起外在的事物，會更好奇事物表象背後的邏輯運作，比如看到漂亮的攝影，我不只會欣賞表象的美感，還會直覺地去思考導演與攝影師如何選擇攝影機運動、分鏡如何設計、攝影師如何配合演員的表演等等。

在寫程式時，知道自己寫了什麼並理解背後運作的邏輯是很重要的能力，而我有對事情追根究柢的習慣，相信這點能幫助我主動釐清程式背後運作的邏輯。


  • 喜歡獨立解決問題，但也樂於與人合作

在從事影像工作的過程中，我發現自己在編劇、剪輯等獨立作業時非常容易進入心流，而我在學習程式時，也再次確認自己享受獨立解決問題的工作型態（更容易進入心流）；另外，曾為影像工作者的我，也深知團隊合作帶來的影響力，因此我也樂於與優秀的夥伴協作，為產品帶來更大的價值。

能夠獨立解決問題又能與人團隊合作是工程師需要的特質，而這也是我認為自己適合成為軟體工程師的原因之一。


在透過自學確認自己有寫程式的熱忱，並剖析自己有許多特質適合擔任軟體工程師後，我便向公司提出離職；即將在三月底離職，四月開始全職投入轉職為軟體工程師，希望能有機會加入 WeHelp，透過實作導向的訓練模式，成功轉職為軟體工程師。
